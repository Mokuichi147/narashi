# モデル比較ベンチマーク (メモ)

narashi の表記ゆれ統合で使う埋め込みモデルを、同じ「物差し」で横並び比較した記録です。
モデルを差し替えるたびにこの手順 (`.claude/skills/model-eval`) を再実行し、本表を更新してください。

## 物差し

- **評価データ**: `tests/data/glossary.txt`(日本語の表記ゆれ + 日英中のクロス言語同義 + 難しい単独語)
  - 同一グループ内ペア = 統合すべき(正例)、異グループ間ペア = 統合すべきでない(負例)
- **実行**: `cargo run --example benchmark -- <model> [threshold]`(指標ロジックは `src/eval.rs`)
- **既定閾値**: `DEFAULT_THRESHOLD = 70.0`
- 指標の意味:
  - **F1@70 / P@70 / R@70**: 既定閾値での分類精度(推移閉包なしの素のペア判定)
  - **best-F1 / best-thr**: 分類 F1 を最大化する閾値とその値(閾値非依存の上限性能)
  - **margin**: `正例min − 負例max`。0 に近いほど正例・負例の分離が良い(完全分離なら正)
  - **clusterF1**: `normalize`(union-find の推移閉包込み)の実出力と正解グループのペア F1 = **実運用の挙動**
  - **per_text**: 1 用語あたり埋め込み時間(load 時間はサンドボックスのコールドスタート依存のため speed は per_text で比較)

### 1 刻みスイープでの真のピーク(grid 外の取りこぼし補正)

⚠️ 上の **5 刻み表は grid 外にピークを持つモデルを系統的に過小評価する**(推移閉包で clusterF1 が
離散的に動くため、ピークが 71・72・89 等の grid 外に落ちる)。`cargo run --example fine_sweep -- <model>`
で **1 刻み(40〜95)** に測り直したのが下表。各モデルの本当のピークと最適閾値を示す。

| モデル | **1 刻みピーク** | 最適閾値 | (参考)5 刻み値 | ピーク時 P / R |
|---|---:|---:|---:|---:|
| **gte-multilingual-base** | **0.682** | 70 | 0.682 | 0.782 / 0.604 |
| e5-large | 0.644 | 72 | 0.587 | 1.000 / 0.475 |
| paraphrase-MiniLM-L12 | 0.626 | 89 | 0.606 | 0.718 / 0.554 |
| e5-small | 0.616 | 71 | 0.584 | 1.000 / 0.446 |

→ **重要(検証で判明した訂正)**: 1 刻みで測り直すと、5 刻み表が示唆した差は縮む。**gte は依然トップ
(0.682)だが、e5-large(0.644)との差は約 0.04** で、5 刻み表の見かけ上の差(約 0.10)ほど大きくない。
e5-small・paraphrase も 5 刻み値より高い(0.616 / 0.626)。それでも gte は **最良の単一動作点を、しかも
既定閾値 70 ちょうどで** 達成する点が依然強み(他モデルのピークは 72・89・71 と grid 外)。

→ **gte のピークは鋭い**。1 刻みで見ると 70 が頂点で、隣接は 69→0.652 / 71→0.674。ただし **64〜78 の
広い帯で 0.63〜0.68** を維持し、この帯はおおむね他モデルのピーク以上。E5 系は閾値 70 付近に鋭い
ピークを持ち、超えると適合率は上がる(75 で P=1.0)が再現率が急落する。paraphrase 系はピークが
高閾値側(89)にある。

## 別系統モデル(多言語特化でないベースライン)

「全く別のモデル」も同じ物差しで測った。いずれも日本語中心の本データでは大きく劣り、**多言語特化の
学習が本質的に効いている**ことを裏づける。

| モデル | 系統 | best-F1 | clusterF1@70 | per_text | 備考 |
|---|---|---:|---:|---:|---|
| bge-small-zh-v1.5 | BGE(中国語特化) | 0.204 | 0.128 | 0.51ms | 中国語語彙には反応するが日本語は学習外 |
| all-MiniLM-L6-v2 | 英語 SentenceTransformers | 0.095 | 0.084 | 1.00ms | 英語専用。日中はほぼ識別不能 |
| clip-ViT-B-32-text | CLIP(対照学習・別アーキ) | 0.085 | 0.072 | 2.22ms | 全テキストが高類似に潰れ(負例max 99.4)、1 クラスタに統合してしまう |

## 所見

- **精度最優先なら gte-multilingual-base が最良**。既定の閾値 70(校正済み `cos_baseline=0.42`)で
  clusterF1 0.682 と全モデル中で最高。1 刻みスイープで各モデルの真のピークと比べても首位だが、差は
  e5-large(0.644)に対し約 0.04・e5-small(0.616)に対し約 0.07 で、**5 刻み表の見かけほど大きくない**
  (下の「1 刻みスイープ」参照)。強みは、最良の単一動作点を **既定閾値 70 ちょうどで** 高 Precision
  (P@70=0.964、誤統合 2 件)のまま達成し、再現率(0.535)も底上げできること。コストは e5-small の
  約 3 倍の推論時間(5.4ms)+ 1.2GB(fp32 単一 ONNX)。組み込みカタログに無いため fastembed の
  `try_new_from_user_defined` でユーザー定義モデルとして読み込む(`--model gte`)。
- **軽量・最速重視なら e5-small が依然有力**。既定閾値での適合率が最高(0.971 ≒ 誤統合がほぼ無い)、
  最速級(1.7ms / 384次元)。1 刻みピーク 0.616(@71)は gte(0.682)に約 0.07 譲るが、速度・サイズ
  (約 0.45GB)が効く用途では好バランス。表記ゆれ統合では「異なる語を誤って統合する(=データ破壊)」方が
  取りこぼしより痛いため、保守的で高精度。**既定の gte が重い/遅い場合の軽量な代替**として最有力。
- **閾値を調整できるなら paraphrase-MiniLM @89 も候補**(1 刻みピーク 0.626、最速 1.28ms)。ただし
  baseline 0.30 のままでは最適点が 89 と高く、運用閾値の再設定が前提。gte には僅差で及ばない。
- **e5-large は E5 系の上限で gte に肉薄**。1 刻みピーク 0.644(@72、P=1.0 R=0.475)は gte(0.682)に
  次ぐ 2 位で、5 刻み表の 0.587 は過小評価だった。best-F1(0.646)・分離 margin(−40.7)も E5 系で最良。
  ただしコストは **約 8 倍の推論時間**(13.6ms)+ 2.2GB と最重量で、精度・速度・サイズの総合では gte
  (0.682 / 5.4ms / 1.2GB)に劣後する。
- **e5-base は劣後**(全閾値で e5-small 以下、かつ約 3 倍遅い)。採用理由なし。
- **別系統(BGE-zh / 英語 MiniLM / CLIP)は本データでは不適**。多言語(特に日本語)対応が必須条件。
- **bge-m3(BAAI, 1024次元)は今回見送り**。多言語性能は有力だが、配布 ONNX(Xenova/bge-m3)が外部重み
  `model.onnx_data`(約 2.2GB)を別ファイルに持ち、fastembed のユーザー定義経路は ONNX をバイト列から
  `commit_from_memory` で読むため外部重みを解決できない(単一ファイル ≤2GB の protobuf 制約を超える)。
  単一ファイル版が用意できれば再評価の余地あり。
- **校正の含意**: E5 系(`cos_baseline=0.70`)は最適閾値が 64–75 と既定 70 付近、paraphrase 系
  (`cos_baseline=0.30`)は 85–90、gte(`cos_baseline=0.42`)は校正済みで 70。paraphrase 系を既定運用するなら
  baseline 引き上げで最適閾値を寄せる再校正が要る。

## 用途別の選び方

| 重視点 | 推奨 | 閾値 |
|---|---|---|
| 精度最優先(**既定**・やや低速・1.2GB) | **gte-multilingual-base** | 70 |
| 軽量・最速・省サイズ(0.45GB) | e5-small | 70 |
| 精度ピーク × 最速(閾値調整可) | paraphrase-MiniLM-L12 | ~89 |
| 再現率最優先 | paraphrase-mpnet-base | ~87 |

> 測定環境: CPU(ONNX Runtime)。`per_text` は相対比較用で、ハードウェアにより絶対値は変動する。
> 別系統 3 種は多言語性能のベースライン(下限)確認用。
