# モデル比較ベンチマーク (メモ)

narashi の表記ゆれ統合で使う埋め込みモデルを、同じ「物差し」で横並び比較した記録です。
モデルを差し替えるたびにこの手順 (`.claude/skills/model-eval`) を再実行し、本表を更新してください。

## 物差し

- **評価データ**: `tests/data/glossary.txt`(日本語の表記ゆれ + 日英中のクロス言語同義 + 難しい単独語)
  - 同一グループ内ペア = 統合すべき(正例)、異グループ間ペア = 統合すべきでない(負例)
- **実行**: `cargo run --example benchmark -- <model> [threshold]`(指標ロジックは `src/eval.rs`)
- **既定閾値**: `DEFAULT_THRESHOLD = 70.0`(CLI のデフォルト値)
- **モデル間比較は最適閾値で行う**: 固定の 70.0 での横並び比較は校正(`cos_baseline`)依存で各モデルの最適動作点を反映しないため意味が薄い。本記録では best-thr / clusterF1 真ピーク(=各モデルの最適閾値での値)で比較する
- 指標の意味:
  - **best-F1 / best-thr**: 分類 F1 を最大化する閾値とその値(各モデルを最適閾値で測った素ペア性能の上限)
  - **margin**: `正例min − 負例max`。0 に近いほど正例・負例の分離が良い(完全分離なら正)
  - **clusterF1(真ピーク)**: `normalize`(union-find の推移閉包込み)の実出力と正解グループのペア F1 = **実運用の挙動**。各モデルの最適閾値(1 刻みスイープのピーク)で比較する
  - **誤統合(false merges)**: `normalize` の推移閉包込みの実出力で、**異グループの語を同一クラスタにしてしまったペア数**(= データ破壊)。表記ゆれ統合では誤統合は取りこぼしより痛いため、clusterF1 が同程度なら誤統合が少ないモデルを選ぶ。**モデル間比較は clusterF1 と同様に各モデルの真ピーク閾値で行う**(固定閾値 70 はモデルごとにピークが異なり、ピーク外では再現率が落ちて誤統合が見かけ上減るため比較として無意味)。`fine_sweep` の各行に誤統合件数が出るので、真ピーク行の値を採る。`benchmark` も既定閾値での件数・具体例と各スイープ行の件数を出力する
  - **per_text**: 1 用語あたり埋め込み時間(load 時間はサンドボックスのコールドスタート依存のため speed は per_text で比較)
  - **最難負例 / 最難正例(閾値非依存)**: 飽和対策で追加した境界指標(下の「ベンチマークの飽和対策」参照)。`benchmark` / `candle_eval` の出力末尾に、誤統合しやすい順の負例ペア(= 適合率の境界)と取りこぼしやすい順の正例ペア(= 再現率の境界)を上位 10 件表示する。**clusterF1 が上位で頭打ちになっても、ここのスコアは上位モデルほど良く(負例は低く・正例は高く)分離が続く**ため、最上位モデル同士の比較の主軸にする

## ベンチマークの飽和対策(データセット v2)

これまでの用語集(**v1: 74 語 / 26 グループ / 2701 ペア**)は、最上位モデルが容易に上限へ達して
モデル間の優劣を分離できなくなった(= 飽和)。代表例として **Qwen3-Embedding-4B が clusterF1 真ピーク
0.964・P=1.000・誤統合 0 件**に達しており、これ以上高性能なモデルが来ても差が出ない。そこで用語集を
**v2: 153 語 / 67 グループ / 11628 ペア**へ拡張し、最上位モデルでも誤りうる「難所」を意図的に増やした。

### 何を難しくしたか

- **難正例(再現率の難所)**: 表層がほとんど重ならない多言語同義語を 10 グループ追加。略語・外来語・
  漢語・英語・中国語が入り混じり、文字の重なりに頼れない。例: `便所 / トイレ / お手洗い / restroom / 厕所`、
  `医者 / 医師 / ドクター / doctor / 医生`、`じゃがいも / ポテト / 馬鈴薯 / potato / 土豆`。強モデルでも取りこぼしうる。
- **難負例(適合率の難所)**: 埋め込み空間で近接するが統合してはいけない語を単独グループで追加。3 種:
  - **共下位語**(同カテゴリの兄弟語): 季節 `春/夏/秋/冬`・色 `赤/青/黄色`・曜日・方角・兄弟・飲料
  - **紛らわしい異義語**(字面が近い別語): `科学/化学`・`保証/保障`・`関心/感心`・`公園/公演`
  - **対義語**(同じ話題で極性だけ反転し空間上で近い): `増加/減少`・`拡大/縮小`

### 新しい比較指標(閾値非依存・希釈に強い)

clusterF1(ペア F1)は易しい負例が大量にあると適合率が見かけ上高くなり(希釈)、上位で差が出にくい。
そこで `src/eval.rs` に**閾値に依存しない境界指標**を追加した:

- **最難負例**: 異グループなのにスコアが高い(最も誤統合しやすい)負例ペアの上位。**上位モデルほど低い**。
- **最難正例**: 同グループなのにスコアが低い(最も取りこぼしやすい)正例ペアの下位。**上位モデルほど高い**。

モデル比較は (1) clusterF1 真ピーク、(2) 真ピーク時の誤統合件数、(3) 最難負例の最高スコア(低いほど良い)、
(4) 最難正例の最低スコア(高いほど良い)で行う。(3)(4) は clusterF1 が飽和しても分離が続くため、
**より高性能なモデルの判定**に使える。

### 効果の実測(qwen3-embedding-0.6b で v1→v2 を比較)

| 指標 | v1(74 語 / 2701 ペア) | v2(153 語 / 11628 ペア) |
|---|---:|---:|
| clusterF1 真ピーク | 0.748 @71 | 0.764 @71 |
| ピーク時 P / R | 0.984 / 0.604 | **0.926** / 0.649 |
| 誤統合(真ピーク時) | **1** | **10** |
| 最難負例の最高スコア | —(v1 未計測) | **81.2**(減少⇔縮小) |
| 最難正例の最低スコア | —(v1 未計測) | **33.5**(人参⇔carrot) |

> clusterF1 の**値自体は希釈でほとんど動かない(0.748→0.764)**が、**真ピーク時の誤統合が 1→10 件に増え、
> 適合率が 0.984→0.926 に低下**した。つまり「最上位でも誤らない」状態が崩れ、難負例で実際に誤るように
> なった。v2 で qwen3-0.6b が誤統合した難負例(@70)は `減少⇔縮小`(81.2)・`保証⇔保障`(79.0)・
> `増加⇔拡大`(78.0)・`火曜日⇔水曜日`(77.0)など、取りこぼした難正例は `人参⇔carrot`(33.5)・
> `台所⇔kitchen`(35.8)・`給与⇔salary`(36.8)など。これらの**最難ペアのスコアは閾値や校正に依存しない
> 連続値**なので、Qwen3-4B のように clusterF1 が飽和するモデル同士でも優劣を分離できる。

### v2 への再計測について

下の詳細表(素性能・5 刻みスイープ・1 刻み真ピーク)は **v1(74 語 / 2701 ペア)で測った旧値**。
v2 では各モデルを `.claude/skills/model-eval` の手順で再計測してこの節を更新する(ONNX 勢は `benchmark` /
`fine_sweep`、Candle 勢は ORT 不要の `candle_eval`)。再計測時は **clusterF1 だけでなく、真ピーク時の誤統合件数と
最難負例・最難正例のスコアを必ず併記**し、最上位モデル同士の分離を確認すること。

> **bge-m3(既定)** (`BAAI/bge-m3`, XLM-RoBERTa-large 系・1024次元・CLS プーリング, MIT) は fastembed の
> 組み込みカタログには無く、`Xenova/bge-m3` の **fp16 ONNX(単一ファイル・約 1.06GB)** を
> `try_new_from_user_defined` で読み込む**ユーザー定義モデル**として追加した(`--model bge-m3`)。fp32 は
> 外部重み付きで単一ファイル化できないが、fp16 版は単一ファイルのため `commit_from_memory` 経由で読める。
> 100+ 言語対応で日本語も対象。`cos_baseline=0.072` に校正し、clusterF1 真ピークを既定閾値 70 ちょうどに
> 合わせている。**clusterF1 真ピーク 0.725 は全モデル中で最高**(従来既定 gte 0.682・granite-278m 0.705 を
> 上回る)で、しかも**ピーク時適合率 P=0.983・真ピーク時の誤統合わずか 1 件**(例: りんご⇔たまご)と
> **最も誤統合が少ない**。完全一致グループも 18/26 と最多。granite-278m のような「高 clusterF1 だが誤統合が
> 多い」トレードオフが無く、**精度(clusterF1)・誤統合の少なさ・サイズ(1.06GB < gte 1.2GB)のすべてで
> gte を上回る**。唯一の弱点は推論速度で、1024次元 + fp16(CPU)のため同一セッション比較で gte の約 3 倍
> (≈16ms/語 対 ≈5ms/語)。バッチ用途では問題にならないため、**要望(大型でも高精度)に合致する本命として
> 既定に採用**した。速度重視なら `--model gte` へ切り替えられる。
>
> **gte-multilingual-base** (Alibaba-NLP, 768次元, CLS プーリング, Apache 2.0) は
> `onnx-community/gte-multilingual-base` の ONNX(単一ファイル・約 1.2GB)を `try_new_from_user_defined` で
> 読み込む**ユーザー定義モデル**(`--model gte`)。`cos_baseline=0.42` に校正し、clusterF1 のピークを既定
> 閾値 70 に合わせている。clusterF1 真ピーク 0.682 は bge-m3(0.725)・granite-278m(0.705)に次ぐ 3 位だが、
> **純粋な分類性能 best-F1=0.776 は全モデル中で最良**で、素ペア誤統合(FP=2)も少ない。推論は bge-m3 の
> 約 1/3(≈5ms/語)と軽快なため、**速度重視時の代替の第一候補**に位置づける。
>
> **granite-278m-multilingual** (`ibm-granite/granite-embedding-278m-multilingual`, IBM Granite
> Embedding R1, XLM-RoBERTa 系・768次元・CLS プーリング, Apache 2.0) は `model.onnx`(単一ファイル・
> 約 1.1GB・外部重み無し)を同じユーザー定義経路で追加(`--model granite`)。200+ 言語のうち日本語は明示的な
> 学習対象。`cos_baseline=0.44` に校正し、clusterF1 真ピーク **0.705**(全候補中で最高)を既定閾値 70 ちょうど

> が、**真ピーク時の誤統合が 24 件と gte(17 件)より多く、素ペアでは FP=19 件と gte(2 件)の約 10 倍**。
> 「猫=犬」のようなデータ破壊型の誤りが増えるため**既定には採用せず、選択肢として追加**するに留めた。
> clusterF1 やクロス言語の再現率を最優先する用途では `--model granite` が有力。
>
> **granite 系のその他**(`--model granite-107m` / `granite-278m` / `granite-311m-r2` / `granite-97m-r2`、
> いずれも Apache 2.0・単一ファイル ONNX・CLS プーリング・プレフィックス無し): **granite-311m-multilingual-r2**

>

> でない素の **bge-m3 が 0.725** と大差で上回り、検索特化の再学習が本タスクには逆効果と分かる。`UserModel`
> として残し eval から選べる(`--example benchmark -- arctic-l`)が CLI 既定の選択肢には出さない。
>
> **multilingual-e5-large-instruct【見送り・Candle】** (`intfloat/multilingual-e5-large-instruct`,
> XLM-RoBERTa-large 系の指示対応 E5・1024次元・Mean プーリング, MIT) は **Candle バックエンドの実例として
> 評価**した(`--model e5-instruct`)。この HF リポジトリの ONNX は外部重み付き(`onnx/model.onnx` +
> `model.onnx_data`)で fastembed の単一ファイル経路では読めず**従来の ONNX 経路では評価できなかった**が、
> Candle が safetensors(fp16 単一・約 1.12GB)を直接読み込むことで評価できた。指示プレフィックス
> (`"Instruct: Retrieve semantically similar text.\nQuery: "`)を両テキストに付与して対称利用し、
> `cos_baseline=0.67` に校正して clusterF1 真ピークを既定閾値 70 に合わせている。結果は **clusterF1 真ピーク
> 0.670**(分類 best-F1=0.714)で、gte(0.682)を下回り distiluse(0.667)と同程度。**ただし真ピーク時の
> 誤統合が 38 件と全評価モデル中で最多**(bge-m3 1・distiluse 3・gte 17・granite-278m 24)で、ピーク時
> 適合率も P=0.648 と最低。指示プレフィックスで全体のコサインが高帯域に寄り、推移閉包で「猫⇔犬」「猫⇔みず」
> のような異義語まで巻き込む大クラスタが生じる(検索向け非対称学習を対称類似度に流用した弊害)。加えて
> 推論は Candle CPU で ≈220–270ms/語と桁違いに遅い。精度・安全性・速度のいずれでも既存の ONNX モデルに
> 劣るため**既定には採用せず見送り**、`UserModel` として残し **ONNX 非依存環境(Candle 単独ビルド)の
> 選択肢**兼バックエンドの実例に留める(`--model e5-instruct` / `--example benchmark -- e5-instruct`)。
>
> **Qwen3-Embedding-0.6B【Candle・精度枠】** (`Qwen/Qwen3-Embedding-0.6B`, Qwen3 デコーダ系・1024次元・
> **last-token プーリング**, Apache 2.0) は **Candle バックエンドで last-token プーリングを実装して評価**した
> (`--model qwen3`)。**last-token プーリングは fastembed(Cls/Mean のみ)が非対応で従来は評価できなかった**
> (上の「サイズ無制限調査」で構造的な壁とした制約)が、Candle が safetensors(bf16 単一・約 1.19GB)を直接読み、
> tokenizer が末尾へ付与する EOS の隠れ状態を取り出すことで初めて評価できた。指示プレフィックス
> (`"Instruct: Retrieve semantically similar text.\nQuery:"`)を両テキストに付与して対称利用し、`cos_baseline=0.69`
> に校正して clusterF1 真ピークを既定閾値 70 付近に合わせている。結果は **clusterF1 真ピーク 0.748(全評価モデル中で
> 最高・既定 bge-m3 の 0.725 を上回る)** をピーク時 P=0.984・**誤統合わずか 1 件**で達成し、精度・安全性ともにトップ。
> 完全一致グループも 18/26 と最多級。「高 clusterF1 は誤統合増と引き換え」というトレードオフを破る点は bge-m3 と
> 同様で、本データでは bge-m3 を精度で上回った。**唯一にして決定的な弱点は速度**で、Qwen3 はデコーダ系のうえ
> candle 0.9 の制約(KvCache のリセット API が非公開・バッチ>1 の因果マスク broadcast が CPU で添字エラー)から
> **1 件ずつモデルを作り直して前向き計算**する必要があり、推論は Candle CPU で ≈200ms/語(bge-m3 の約 12 倍)。
> このため **既定は速度重視で bge-m3 のまま**とし、Qwen3-0.6B は **精度重視/ONNX 非依存環境(Candle 単独
> ビルド)の選択肢**として提供する(`--model qwen3`、Candle 単独ビルドでは既定)。
>
> **Qwen3-Embedding-4B【Candle・精度上限】** (`Qwen/Qwen3-Embedding-4B`, Qwen3 デコーダ系・2560次元・36層・
> **last-token プーリング**, Apache 2.0) は 0.6B と同一アーキの大型版。重みは分割保存(2 シャード)で、CPU の
> matmul が bf16 非対応のため **f16 で読み込む(約 8GB)**。結果は **clusterF1 真ピーク 0.964 を既定閾値 70
> ちょうどで達成し(P=1.000・R=0.931・誤統合 0 件)、全評価モデル中で群を抜く最高精度**。完全一致グループは
> 24/26。ピーク帯も 66→0.949・70→0.964・77→0.942 と広く安定し、適合率は 70 以上で常に 1.000(誤統合ゼロ)。
> 0.6B(0.748)から再現率が 0.604→0.931 へ大きく伸び、誤統合ゼロのまま取りこぼしが激減したのが効いている。
> **唯一にして決定的な弱点は速度**で、4B + f16(CPU の f16 matmul は f32 より遅い)を 1 件ずつ前向き計算する
> ため **≈3.9 秒/語**(bge-m3 の約 240 倍・0.6B の約 20 倍)、ロード約 48 秒・RAM 約 8GB。**GPU・バッチ・
> オフライン等で速度を許容できる「精度を最優先する用途」向け**に `--model qwen3-4b` で提供する。
> **8B**(`Qwen/Qwen3-Embedding-8B`・4096次元)は同じ読み込み経路(分割・f16)で `--model qwen3-8b` から選べるが、
> f16 でも約 16GB RAM を要し本評価環境(15GB)では未測定。十分な RAM の環境なら 4B をさらに上回る可能性がある。

### 1 刻みスイープでの真のピーク(モデル間比較の主表)

全モデルを大きく引き離す。**弱点は速度/RAM のみ**(Candle CPU・f16・1 件ずつ前向き計算・≈3.9 秒/語・約 8GB RAM)で、
GPU・バッチ・オフライン等で速度を許容できる精度最優先用途向け(`--model qwen3-4b`)。中間の Qwen3-0.6B(0.748・
P=0.984・誤統合 1 件・≈200ms/語)は ONNX 非依存環境(Candle 単独ビルドでは既定)の最有力(`--model qwen3`)。

サイズも 1.06GB と gte(1.2GB)より小さい。速度を最優先する場合のみ gte(0.682・誤統合 17 件・約 1/3 の
推論時間)を `--model gte` で選ぶ。

→ **granite-278m は clusterF1 2 位(0.705)だが誤統合 24 件**で、bge-m3 に精度でも安全性でも劣る。
高再現率(R=0.673)が必要な特殊用途の選択肢(`--model granite`)に留める。

持ち 64〜78 で 0.63〜0.68 と安定。E5 系は閾値 70 付近に鋭いピークを持ち、超えると適合率は上がる
(75 で P=1.0)が再現率が急落する。paraphrase 系はピークが高閾値側(89)にある。

## 評価できなかった候補(配布形式の制約)

fastembed のユーザー定義経路は ONNX をバイト列から `commit_from_memory` で読むため、**単一ファイル
ONNX(≤2GB・外部重み `model.onnx_data` なし)** しか読み込めない。従来既定 gte を超える/軽量枠でより良い
多言語モデルがないか調査した。**重要な突破口**: 当初「外部重み(fp32 で約 2.2GB)で評価不可」としていた
大型モデルでも、**fp16 版の単一ファイル ONNX(≤2GB)が配布されていれば評価できる**。実際に
`Xenova/bge-m3` の fp16(1.06GB)で **bge-m3 を評価し、clusterF1 0.725 で全モデル中最高・誤統合も最小**と
判明したため**既定に採用**した。同様に大型の snowflake-arctic-embed-l-v2.0(fp16)も評価したが検索特化で
低調(見送り)。

**さらなる突破口 — Candle バックエンド**: その後追加した Candle バックエンド(`config.json` の `model_type` で
アーキを判定し safetensors を直接読む。**分割保存・f16 にも対応**)により、**ONNX 配布形式やプーリング方式・
サイズの制約で従来評価できなかったモデルも評価可能**になった。これで(1)外部重み付き ONNX しか無い
`e5-large-instruct`(XLM-RoBERTa・見送り)、(2)last-token プーリングが必要な `Qwen3-Embedding`(Qwen3 デコーダ)を
評価でき、**Qwen3-0.6B で 0.748、さらに 4B(f16・約 8GB)で clusterF1 0.964(P=1.000・誤統合 0 件)と全モデル
中ぶっちぎりの最高精度**と判明した(いずれも採用)。残る制約は速度/RAM(4B は ≈3.9 秒/語、8B は約 16GB RAM で
本環境では未測定)・アーキ実装・トークナイザ(ruri-base は MeCab 依存)・ライセンス(jina-v3)。

| jina-embeddings-v3 (572M) | ⚠ fp16 単一あり但し CC-BY-NC | ライセンス非寛容(NC)のため対象外(Candle でも読めるが採用不可) |
| BAAI/bge-multilingual-gemma2 (9B) | ⚠ last-token + Gemma ライセンス + サイズ超過 | ライセンス非寛容・サイズ超過で対象外 |
| cl-nagoya/ruri-base (日本語特化, Apache 2.0) | ❌ BertJapaneseTokenizer(MeCab/unidic)依存 | BertModel 自体は Candle で読めるが、**トークナイザが MeCab 形態素解析前提で `tokenizer.json` 非配布**。純 Rust の `tokenizers` では再現できず、形態素解析器(lindera 等・大きな辞書依存)が別途必要なため見送り |

**モデルサイズ無制限での高性能モデル調査(まとめ)**: 「サイズに制限をかけずより高性能なモデルを」という
方針で大型・高 MTEB 候補を洗い直した。fastembed の user-defined 経路は **単一ファイル ONNX(≤2GB)+ Cls/Mean
プーリング** が必須で、**現行の SOTA 級大型多言語モデルの多くは last-token プーリング採用**(Qwen3-Embedding、
gte-Qwen2、e5-mistral-7b、SFR-Embedding、NV-Embed、bge-multilingual-gemma2 等)で **fastembed では実行不可**
という構造的な壁があった。**その後追加した Candle バックエンドが last-token プーリング・safetensors 直読み・
分割保存・f16 に対応してこの壁を破った**。実際に **Qwen3-Embedding を Candle で評価し、0.6B で clusterF1 0.748、
さらに 4B(f16・約 8GB)で clusterF1 0.964(P=1.000・誤統合 0 件)と全モデル中ぶっちぎりの最高精度**を確認した
(いずれも採用)。残る制約は(1)**速度/RAM**——4B は ≈3.9 秒/語と極めて低速、8B は f16 でも約 16GB RAM で
本環境(15GB)では未測定、gte-Qwen2-7B・e5-mistral-7b は 7〜8B 級でさらに非現実的。Cls/Mean の ONNX
大型枠でも runnable な多言語エンコーダは **bge-m3(XLM-RoBERTa-large)が事実上の上限**——と(2)**ライセンス**
(jina-v3 は CC-BY-NC、gemma2 系は非寛容)、(3)**トークナイザ**(ruri-base は MeCab 依存)。なお Candle 版
e5-instruct(0.670)のように **instruct/検索チューニング系は短い対称語の統合では埋め込み幾何が崩れる**傾向が
あり、「大型・高 MTEB ＝ 本タスクで高性能」ではない一方、**素の対称埋め込みである
Qwen3-Embedding は本タスクに適合**した。

**結論**: fp16 単一ファイル ONNX に加え **Candle バックエンド(safetensors 直読み + last-token 等のプーリング +
分割保存 + f16 + `model_type` でアーキ判定)** により、配布形式・プーリング方式の制約で評価できなかった
モデルまで評価できるようになった。その結果、**精度の絶対最高は `Qwen3-Embedding-4B`(Candle・Apache 2.0・
clusterF1 真ピーク 0.964・P=1.000・誤統合 0 件)**で全モデルを大きく引き離した。ただし ≈3.9 秒/語・約 8GB RAM と
極めて重く、`Qwen3-Embedding-0.6B`(0.748・≈200ms/語)も Candle CPU で遅い。そのため **総合の既定は ONNX で
高速な `bge-m3`(clusterF1 0.725・誤統合 1 件・≈16ms/語)** に据える。bge-m3 は granite-278m のような
「高 clusterF1 だが誤統合が多い(24 件)」トレードオフが無く、従来既定 gte(0.682 / 誤統合 17 件)を精度・
安全性・サイズ(1.06GB < gte 1.2GB)で上回る。速度重視時の代替は **gte**(0.682・約 1/3 の推論時間)、軽量枠は
**distiluse**(0.667・最小サイズ・最速級)、**精度を絶対最優先(GPU/バッチ可)なら qwen3-4b(0.964)**、その中間が
**qwen3(0.6B・0.748)**。同じ BGE-M3 系でも検索特化の arctic-l-v2.0 は 0.510 と低調で、検索特化の再学習は
本タスクには逆効果と確認した。

## 所見

- **精度の絶対最高は Qwen3-Embedding-4B(Candle・精度上限枠)**。clusterF1 真ピーク **0.964(P=1.000・R=0.931・
  誤統合 0 件)を既定閾値 70 ちょうどで達成**し、全モデルを大きく引き離す。完全一致グループ 24/26。0.6B
  (0.748)から再現率が 0.604→0.931 へ伸び、誤統合ゼロのまま取りこぼしが激減した。**弱点は速度/RAM のみ**で、
  4B + f16(CPU の f16 matmul は遅い)を 1 件ずつ前向き計算するため ≈3.9 秒/語(bge-m3 の約 240 倍)・約 8GB RAM。
  GPU・バッチ・オフライン等で速度を許容できる**精度最優先用途**向け(`--model qwen3-4b`、Apache 2.0)。8B は同経路で
  `--model qwen3-8b` から選べるが約 16GB RAM 必須で本環境では未測定。
- **その中間が Qwen3-Embedding-0.6B(Candle・精度枠)**。clusterF1 真ピーク 0.748・ピーク時 P=0.984・誤統合 1 件で、
  bge-m3(0.725)を精度で上回る。Qwen3 デコーダ + last-token を Candle に実装(`--model qwen3`、Apache 2.0)。
  candle 0.9 の制約(KvCache リセット非公開・バッチ broadcast バグ)から 1 件ずつモデルを作り直して前向き計算する
  ため Candle CPU で ≈200ms/語(bge-m3 の約 12 倍)。**ONNX 非依存環境(Candle 単独ビルドでは既定)の最有力**。
  バッチ前向き計算や KvCache リセットが candle 側で解決すれば高速化の余地あり。
- **既定は bge-m3(精度と速度のバランス・ONNX で高速)**。clusterF1 真ピーク **0.725 は Qwen3 に次ぐ 2 位**で、
  ピーク時 P=0.983・真ピーク時の誤統合わずか 1 件と**誤統合は最小タイ**。完全一致グループ 18/26 も最多。
  Qwen3(0.748)に精度で約 0.02 譲るが、**ONNX で約 12 倍高速(≈16ms/語)**なため総合の既定に最適。
  「高 clusterF1 は誤統合増と引き換え」というトレードオフ(granite-278m は 0.705 だが誤統合 24 件)を破り、
  精度・安全性・サイズ(1.06GB)のすべてで従来既定 gte を上回る。`BAAI/bge-m3` の fp16 単一 ONNX
  (`Xenova/bge-m3`)を `try_new_from_user_defined` で読み込む(`--model bge-m3`、`cos_baseline=0.072` 校正で
  最適閾値は CLI 既定 70)。唯一の弱点は推論速度で、1024次元 + fp16(CPU)のため gte の約 3 倍(≈16ms/語)。
- **速度重視の代替は gte-multilingual-base**。clusterF1 真ピーク 0.682 と bge-m3 に約 0.04 譲るが、純粋な
  分類性能 best-F1=0.776 は全モデル中で最良、ピーク時 P=0.782・誤統合 17 件と適合率も高く、推論は bge-m3 の
  約 1/3(≈5ms/語)と軽快。CJK に強い 768次元・約 1.2GB。`--model gte`(`cos_baseline=0.42` 校正で最適閾値
  は CLI 既定 70)。
- **clusterF1/再現率を最優先するなら granite-278m-multilingual(選択肢)**。clusterF1 真ピーク **0.705** で
  bge-m3(0.725)に次ぐ 2 位、ピーク時 P/R=0.739/0.673 と再現率が高い。IBM Granite Embedding R1
  (XLM-RoBERTa 系・Apache 2.0)で 200+ 言語のうち日本語は明示的な学習対象。約 1.1GB。**ただし真ピーク時の
  誤統合が 24 件(bge-m3 1 件・gte 17 件)**と多く「猫=犬」型のデータ破壊が増えるため、再現率最優先の用途の
  選択肢に留める(`--model granite`、`cos_baseline=0.44` 校正で最適閾値は CLI 既定 70)。
- **軽量代替の第一候補は distiluse-multilingual-v2**。clusterF1 真ピーク **0.667** を高適合率
  (ピーク時 P/R=0.945/0.515)で達成。gte(0.682)に約 0.015 譲るだけで、e5-large(0.644)・
  e5-small(0.616)を上回る。**最小サイズ(0.54GB)・最速級(2.45ms)**のため、gte が重い/遅い用途での
  軽量な代替として最有力。`--model distiluse`(`cos_baseline=0.39` 校正で最適閾値は 70)。
- **e5-small もなお有力**(最軽量 0.45GB・最速 1.7ms・384次元)。最適閾値 71 でピーク時 P=1.000 と
  最も保守的。clusterF1 真ピーク 0.616 は distiluse(0.667)に約 0.05 譲るが、サイズ・速度を最優先する
  用途や、誤統合を極限まで避けたい場合に好バランス。表記ゆれ統合では「異なる語を誤って統合する
  (=データ破壊)」方が取りこぼしより痛いため、保守性重視ならこちら。
- **閾値を調整できるなら paraphrase-MiniLM @89 も候補**(1 刻みピーク 0.626、最速 1.28ms)。ただし
  baseline 0.30 のままでは最適点が 89 と高く、運用閾値の再設定が前提。gte には僅差で及ばない。
- **e5-large は E5 系の上限で gte に肉薄**。1 刻みピーク 0.644(@72、P=1.0 R=0.475)は gte(0.682)に
  次ぐ 2 位で、5 刻み表の 0.587 は過小評価だった。best-F1(0.646)・分離 margin(−40.7)も E5 系で最良。
  ただしコストは **約 8 倍の推論時間**(13.6ms)+ 2.2GB と最重量で、精度・速度・サイズの総合では gte
  (0.682 / 5.4ms / 1.2GB)に劣後する。
- **e5-base は劣後**(全閾値で e5-small 以下、かつ約 3 倍遅い)。採用理由なし。
- **granite 系の 278m 以外は採用せず**。granite-311m-r2(R2 最新・1.25GB)は 0.663 で適合率は高い
  (P=0.824)がピークが高閾値側(88)で gte 未満、granite-107m(0.43GB)は 0.652、granite-97m-r2
  (0.39GB)は 0.551 と、いずれも 278m・gte に届かない。コードには `UserModel` として残し eval から
  選べる(`--example benchmark -- granite-107m` 等)が CLI 既定の選択肢には出さない。
- **snowflake-arctic-embed-m-v2.0 / l-v2.0 は採用せず**。いずれも BGE-M3 系を検索向けに再学習したモデルだが
  検索特化のため対称類似度では低調(m-v2.0 は真ピーク 0.512【非推奨】、l-v2.0 は 0.510【見送り】)。素の
  bge-m3(0.725)が大差で上回り、検索特化の再学習が本タスクには逆効果と分かる。l-v2.0 は eval から選べる
  (`--example benchmark -- arctic-l`)が CLI 既定の選択肢には出さない。
- **別系統(BGE-zh / 英語 MiniLM / CLIP)は本データでは不適**。多言語(特に日本語)対応が必須条件。
- **multilingual-e5-large-instruct は見送り(Candle で `--model e5-instruct` として追加)**。外部重み付き
  ONNX で従来は評価できなかったが、Candle が safetensors を直読みして評価(真ピーク 0.670)。gte(0.682)
  未満で、**真ピーク時の誤統合 38 件と全評価モデル中で最多**(指示プレフィックスでコサインが高帯域に寄り
  推移閉包が異義語まで巻き込む)、推論も Candle CPU で ≈250ms/語と最遅級。精度・安全性・速度のいずれでも
  既定に届かないが、auto-download 可のため ONNX 非依存環境(Candle 単独ビルド)の選択肢に残す
  (`--model e5-instruct`、`cos_baseline=0.67`)。
- **Candle が last-token プーリングの壁を破り、Qwen3-Embedding-0.6B が精度トップに**。従来「現行 SOTA 大型の多くは
  last-token 採用で fastembed では実行不可」という構造的な壁があったが、Candle に last-token を実装して
  Qwen3-Embedding-0.6B を評価し clusterF1 0.748(全モデル中最高)を確認(精度枠で採用)。
- **なお対象外/見送りのまま残る候補**: jina-v3〔CC-BY-NC〕・bge-multilingual-gemma2〔非寛容ライセンス〕は
  ライセンス、Qwen3-Embedding-4B/8B・gte-Qwen2-7B・e5-mistral-7b・SFR・NV-Embed〔last-token だが大型〕は
  Candle CPU では非現実的に遅いサイズ、ruri-base は MeCab トークナイザ依存(`tokenizer.json` 非配布)。
  詳細は上の「評価できなかった候補」を参照。
- **校正の含意**: E5 系(`cos_baseline=0.70`)は最適閾値が 64–75 と既定 70 付近、paraphrase 系
  (`cos_baseline=0.30`)は 85–90、gte(`cos_baseline=0.42`)・bge-m3(`cos_baseline=0.072`)は校正済みで 70。
  bge-m3 はコサインが高帯域に強く圧縮されるため baseline を低く(0.072)取って 70 にピークを寄せている。
