# モデル比較ベンチマーク (メモ)

narashi の表記ゆれ統合で使う埋め込みモデルを、同じ「物差し」で横並び比較した記録です。
モデルを差し替えるたびにこの手順 (`.claude/skills/model-eval`) を再実行し、本表を更新してください。

## 物差し

- **評価データ**: `tests/data/glossary.txt`(日本語の表記ゆれ + 日英中のクロス言語同義 + 難しい単独語)
  - 同一グループ内ペア = 統合すべき(正例)、異グループ間ペア = 統合すべきでない(負例)
- **実行**: `cargo run --example benchmark -- <model> [threshold]`(指標ロジックは `src/eval.rs`)
- **既定閾値**: `DEFAULT_THRESHOLD = 70.0`(CLI のデフォルト値)
- **モデル間比較は最適閾値で行う**: 固定の 70.0 での横並び比較は校正(`cos_baseline`)依存で各モデルの最適動作点を反映しないため意味が薄い。本記録では best-thr / clusterF1 真ピーク(=各モデルの最適閾値での値)で比較する
- 指標の意味:
  - **best-F1 / best-thr**: 分類 F1 を最大化する閾値とその値(各モデルを最適閾値で測った素ペア性能の上限)
  - **margin**: `正例min − 負例max`。0 に近いほど正例・負例の分離が良い(完全分離なら正)
  - **clusterF1(真ピーク)**: `normalize`(union-find の推移閉包込み)の実出力と正解グループのペア F1 = **実運用の挙動**。各モデルの最適閾値(1 刻みスイープのピーク)で比較する
  - **誤統合(false merges)**: `normalize` の推移閉包込みの実出力で、**異グループの語を同一クラスタにしてしまったペア数**(= データ破壊)。表記ゆれ統合では誤統合は取りこぼしより痛いため、clusterF1 が同程度なら誤統合が少ないモデルを選ぶ。**モデル間比較は clusterF1 と同様に各モデルの真ピーク閾値で行う**(固定閾値 70 はモデルごとにピークが異なり、ピーク外では再現率が落ちて誤統合が見かけ上減るため比較として無意味)。`fine_sweep` の各行に誤統合件数が出るので、真ピーク行の値を採る。`benchmark` も既定閾値での件数・具体例と各スイープ行の件数を出力する
  - **per_text**: 1 用語あたり埋め込み時間(load 時間はサンドボックスのコールドスタート依存のため speed は per_text で比較)

### 1 刻みスイープでの真のピーク(モデル間比較の主表)

各モデルは最適閾値が異なるため、固定閾値での比較は不公平になる。`cargo run --example fine_sweep -- <model>`
で **1 刻み(40〜95)** に測り直し、各モデルの本当のピーク(clusterF1)とその最適閾値で比較する。これが
実運用挙動でのモデル間比較の主表。

「誤統合(真ピーク時)」は**各モデルの真ピーク閾値**での `normalize` 実出力(推移閉包込み)の誤統合ペア数。
clusterF1 と同じく真ピークで測る(固定閾値 70 はモデルごとにピークが異なり比較として無意味なため使わない)。
clusterF1 と並ぶ採用判断の主要指標(— は真ピーク時誤統合を未測定)。

| モデル | **1 刻みピーク** | 最適閾値 | ピーク時 P / R | 誤統合(真ピーク時) |
|---|---:|---:|---:|---:|
| **bge-m3(既定)** | **0.725** | 70 | **0.983** / 0.574 | **1** |
| granite-278m-multilingual | 0.705 | 70 | 0.739 / 0.673 | 24 |
| gte-multilingual-base | 0.682 | 70 | 0.782 / 0.604 | 17 |

| LaBSE【見送り】 | 0.662 | 89 | 1.000 / 0.495 | — |

> 誤統合は clusterF1 と同じく**各モデルの真ピーク閾値で測る**。arctic-l はその必要性を端的に示す例で、
> 真ピーク(閾値 51)では誤統合 13 件だが、固定の閾値 70 で測るとピーク外で再現率が 0.129 まで落ちるため
> 誤統合 0 件に見えてしまう(統合がほぼ起きないだけで、安全という意味ではない)。固定閾値での横並びは
> モデルごとにピーク位置が異なる以上、比較として成立しない。

> e5-base・mpnet は 1 刻み未測定。5 刻みスイープ表でのピークはそれぞれ 0.508(@65)・0.588(@85)で、
> いずれも上位群に届かない。

→ **重要**: 1 刻みで測り直すと各モデルが最適閾値で評価される。clusterF1 真ピークだけ見ると
**bge-m3 が最高(0.725)**で、granite-278m(0.705)・gte(0.682)・distiluse(0.667)が続く。
なお各モデルの最適閾値は校正(`cos_baseline`)で動かせるため、最適点が 70 か否かは優劣ではない。

→ **bge-m3 は精度と安全性(誤統合の少なさ)を同時に満たす唯一のモデル**。clusterF1 が最高であるだけで
なく、ピーク時適合率 P=0.983・真ピーク時の誤統合わずか 1 件と**最も誤統合が少ない**。これは
「高 clusterF1 は誤統合の増加と引き換え」という従来のトレードオフ(granite-278m は 0.705 だが誤統合 24 件)
を破る結果で、表記ゆれ統合では誤統合(データ破壊)が取りこぼしより痛いという方針に照らしても文句なく
最良。サイズも 1.06GB と gte(1.2GB)より小さい。**唯一の弱点は推論速度(gte の約 3 倍)**だが、バッチ
用途では許容範囲のため**既定に採用**した。速度を最優先する場合のみ gte(0.682・誤統合 17 件・約 1/3 の
推論時間)を `--model gte` で選ぶ。

→ **granite-278m は clusterF1 2 位(0.705)だが誤統合 24 件**で、bge-m3 に精度でも安全性でも劣る。
高再現率(R=0.673)が必要な特殊用途の選択肢(`--model granite`)に留める。

→ **ピーク帯の形**: bge-m3 は 70 が鋭い頂点(0.725)で、65→0.610・70→0.725・75→0.626 と 70 を中心に対称的。
granite-278m は 70 が頂点(0.705)で 65〜78 の広い帯で 0.62〜0.705 を維持。gte も鋭いピーク(70 で 0.682)を
持ち 64〜78 で 0.63〜0.68 と安定。E5 系は閾値 70 付近に鋭いピークを持ち、超えると適合率は上がる
(75 で P=1.0)が再現率が急落する。paraphrase 系はピークが高閾値側(89)にある。

## 評価できなかった候補(配布形式の制約)

fastembed のユーザー定義経路は ONNX をバイト列から `commit_from_memory` で読むため、**単一ファイル
ONNX(≤2GB・外部重み `model.onnx_data` なし)** しか読み込めない。従来既定 gte を超える/軽量枠でより良い
多言語モデルがないか調査した。**重要な突破口**: 当初「外部重み(fp32 で約 2.2GB)で評価不可」としていた
大型モデルでも、**fp16 版の単一ファイル ONNX(≤2GB)が配布されていれば評価できる**。実際に
`Xenova/bge-m3` の fp16(1.06GB)で **bge-m3 を評価し、clusterF1 0.725 で全モデル中最高・誤統合も最小**と
判明したため**既定に採用**した。同様に大型の snowflake-arctic-embed-l-v2.0(fp16)も評価したが検索特化で
低調(見送り)。

| 候補 | 単一ファイル ONNX? | 結果 |
|---|---|---|
| bge-m3 (BAAI, 1024次元) | ✅ fp16 単一 1.06GB(`Xenova/bge-m3`) | 評価実施 → **既定に採用**(`--model bge-m3`)。clusterF1 0.725 で全モデル中最高・誤統合 1 件で最小 |
| granite-embedding-278m-multilingual (R1, 278M) | ✅ 単一 1.1GB | 評価実施 → 選択肢に追加(`--model granite`)。clusterF1 0.705 だが誤統合が多く既定は見送り |
| granite-embedding-311m-multilingual-r2 (R2, 311M) | ✅ 単一 1.25GB | 評価実施 → 見送り(0.663・高閾値ピークで gte 未満) |
| granite-embedding-107m-multilingual (R1, 107M) | ✅ 単一 0.43GB | 評価実施 → 見送り(0.652・gte/distiluse に届かず) |
| granite-embedding-97m-multilingual-r2 (R2, 97M) | ✅ 単一 0.39GB | 評価実施 → 見送り(0.551・本データで振るわず) |
| snowflake-arctic-embed-l-v2.0 (568M) | ✅ fp16 単一 1.06GB(`Snowflake/...`) | 評価実施 → 見送り(0.510・検索特化で対称類似度に不向き) |
| snowflake-arctic-embed-m-v2.0 (305M) | ✅ 単一 1.2GB | 評価実施 → 精度が低く【非推奨】(詳細は上の結果表) |
| distiluse-base-multilingual-cased-v2 (DistilBERT 6層) | ✅ 単一 0.54GB | 評価実施 → 採用(軽量代替の第一候補) |
| LaBSE (BERT, 471M) | ✅ 単一 1.88GB | 評価実施 → 見送り(gte/distiluse に届かず・重い) |
| multilingual-e5-large-instruct (560M) | ❌ 外部重み(fp16 単一版なし) | 評価不可 |
| jina-embeddings-v3 (572M) | ⚠ fp16 単一あり但し CC-BY-NC | ライセンス非寛容(NC)のため対象外 |
| Qwen3-Embedding-0.6B (Apache 2.0) | ⚠ fp16 単一あり但し last-token プーリング | fastembed は Cls/Mean のみで last-token 非対応のため評価不可 |
| cl-nagoya/ruri-base (日本語特化) | ❌ ONNX 未配布(safetensors のみ) | 評価不可 |

**結論**: fp16 単一ファイル ONNX を使うことで、これまで配布形式の制約で評価できなかった大型モデルを評価
できるようになった。その結果 **`bge-m3`(fp16・MIT・1024次元)が clusterF1 真ピーク 0.725 で全候補中最高**、
かつ**真ピーク時の誤統合わずか 1 件・ピーク時 P=0.983 と最も誤統合も少なく**、精度・安全性・サイズ
(1.06GB < gte 1.2GB)のすべてで従来既定 gte(0.682 / 誤統合 17 件)を上回った。granite-278m のような
「高 clusterF1 だが誤統合が多い(24 件)」トレードオフが無いため、**既定を `bge-m3` に変更**した。唯一の
弱点は推論速度(gte の約 3 倍・≈16ms/語)だが、バッチ用途では許容範囲。速度重視時の代替は **gte**
(0.682・約 1/3 の推論時間)、軽量枠は **distiluse**(0.667・最小サイズ・最速級)。同じ BGE-M3 系でも
検索特化の arctic-l-v2.0 は 0.510 と低調で、検索特化の再学習は本タスクには逆効果と確認した。

## 別系統モデル(多言語特化でないベースライン)

「全く別のモデル」も同じ物差しで測った。いずれも日本語中心の本データでは大きく劣り、**多言語特化の
学習が本質的に効いている**ことを裏づける。

| モデル | 系統 | best-F1 | per_text | 備考 |
|---|---|---:|---:|---|
| bge-small-zh-v1.5 | BGE(中国語特化) | 0.204 | 0.51ms | 中国語語彙には反応するが日本語は学習外 |
| all-MiniLM-L6-v2 | 英語 SentenceTransformers | 0.095 | 1.00ms | 英語専用。日中はほぼ識別不能 |
| clip-ViT-B-32-text | CLIP(対照学習・別アーキ) | 0.085 | 2.22ms | 全テキストが高類似に潰れ(負例max 99.4)、1 クラスタに統合してしまう |

## 所見

- **既定は bge-m3(精度と誤統合の少なさを両立)**。clusterF1 真ピーク **0.725 が全モデル中で最高**、かつ
  ピーク時 P=0.983・真ピーク時の誤統合わずか 1 件と**最も誤統合も少ない**。完全一致グループ 18/26 も最多。
  「高 clusterF1 は誤統合増と引き換え」というトレードオフ(granite-278m は 0.705 だが誤統合 24 件)を破り、
  精度・安全性・サイズ(1.06GB)のすべてで従来既定 gte を上回る。`BAAI/bge-m3` の fp16 単一 ONNX
  (`Xenova/bge-m3`)を `try_new_from_user_defined` で読み込む(`--model bge-m3`、`cos_baseline=0.072` 校正で
  最適閾値は CLI 既定 70)。唯一の弱点は推論速度で、1024次元 + fp16(CPU)のため gte の約 3 倍(≈16ms/語)。
- **速度重視の代替は gte-multilingual-base**。clusterF1 真ピーク 0.682 と bge-m3 に約 0.04 譲るが、純粋な
  分類性能 best-F1=0.776 は全モデル中で最良、ピーク時 P=0.782・誤統合 17 件と適合率も高く、推論は bge-m3 の
  約 1/3(≈5ms/語)と軽快。CJK に強い 768次元・約 1.2GB。`--model gte`(`cos_baseline=0.42` 校正で最適閾値
  は CLI 既定 70)。
- **clusterF1/再現率を最優先するなら granite-278m-multilingual(選択肢)**。clusterF1 真ピーク **0.705** で
  bge-m3(0.725)に次ぐ 2 位、ピーク時 P/R=0.739/0.673 と再現率が高い。IBM Granite Embedding R1
  (XLM-RoBERTa 系・Apache 2.0)で 200+ 言語のうち日本語は明示的な学習対象。約 1.1GB。**ただし真ピーク時の
  誤統合が 24 件(bge-m3 1 件・gte 17 件)**と多く「猫=犬」型のデータ破壊が増えるため、再現率最優先の用途の
  選択肢に留める(`--model granite`、`cos_baseline=0.44` 校正で最適閾値は CLI 既定 70)。
- **軽量代替の第一候補は distiluse-multilingual-v2**。clusterF1 真ピーク **0.667** を高適合率
  (ピーク時 P/R=0.945/0.515)で達成。gte(0.682)に約 0.015 譲るだけで、e5-large(0.644)・
  e5-small(0.616)を上回る。**最小サイズ(0.54GB)・最速級(2.45ms)**のため、gte が重い/遅い用途での
  軽量な代替として最有力。`--model distiluse`(`cos_baseline=0.39` 校正で最適閾値は 70)。
- **e5-small もなお有力**(最軽量 0.45GB・最速 1.7ms・384次元)。最適閾値 71 でピーク時 P=1.000 と
  最も保守的。clusterF1 真ピーク 0.616 は distiluse(0.667)に約 0.05 譲るが、サイズ・速度を最優先する
  用途や、誤統合を極限まで避けたい場合に好バランス。表記ゆれ統合では「異なる語を誤って統合する
  (=データ破壊)」方が取りこぼしより痛いため、保守性重視ならこちら。
- **閾値を調整できるなら paraphrase-MiniLM @89 も候補**(1 刻みピーク 0.626、最速 1.28ms)。ただし
  baseline 0.30 のままでは最適点が 89 と高く、運用閾値の再設定が前提。gte には僅差で及ばない。
- **e5-large は E5 系の上限で gte に肉薄**。1 刻みピーク 0.644(@72、P=1.0 R=0.475)は gte(0.682)に
  次ぐ 2 位で、5 刻み表の 0.587 は過小評価だった。best-F1(0.646)・分離 margin(−40.7)も E5 系で最良。
  ただしコストは **約 8 倍の推論時間**(13.6ms)+ 2.2GB と最重量で、精度・速度・サイズの総合では gte
  (0.682 / 5.4ms / 1.2GB)に劣後する。
- **e5-base は劣後**(全閾値で e5-small 以下、かつ約 3 倍遅い)。採用理由なし。
- **granite 系の 278m 以外は採用せず**。granite-311m-r2(R2 最新・1.25GB)は 0.663 で適合率は高い
  (P=0.824)がピークが高閾値側(88)で gte 未満、granite-107m(0.43GB)は 0.652、granite-97m-r2
  (0.39GB)は 0.551 と、いずれも 278m・gte に届かない。コードには `UserModel` として残し eval から
  選べる(`--example benchmark -- granite-107m` 等)が CLI 既定の選択肢には出さない。
- **snowflake-arctic-embed-m-v2.0 / l-v2.0 は採用せず**。いずれも BGE-M3 系を検索向けに再学習したモデルだが
  検索特化のため対称類似度では低調(m-v2.0 は真ピーク 0.512【非推奨】、l-v2.0 は 0.510【見送り】)。素の
  bge-m3(0.725)が大差で上回り、検索特化の再学習が本タスクには逆効果と分かる。l-v2.0 は eval から選べる
  (`--example benchmark -- arctic-l`)が CLI 既定の選択肢には出さない。
- **別系統(BGE-zh / 英語 MiniLM / CLIP)は本データでは不適**。多言語(特に日本語)対応が必須条件。
- **配布形式で評価できなかった/対象外の候補**(e5-large-instruct〔fp16 単一版なし〕/ jina-v3〔CC-BY-NC〕/
  Qwen3-Embedding〔last-token プーリング非対応〕/ ruri-base〔ONNX 未配布〕)は上の「評価できなかった候補」を
  参照。**fp16 単一 ONNX + 寛容ライセンス + Cls/Mean プーリング**が揃えば再評価の余地あり。
- **校正の含意**: E5 系(`cos_baseline=0.70`)は最適閾値が 64–75 と既定 70 付近、paraphrase 系
  (`cos_baseline=0.30`)は 85–90、gte(`cos_baseline=0.42`)・bge-m3(`cos_baseline=0.072`)は校正済みで 70。
  bge-m3 はコサインが高帯域に強く圧縮されるため baseline を低く(0.072)取って 70 にピークを寄せている。
