# モデル比較ベンチマーク (メモ)

narashi の表記ゆれ統合で使う埋め込みモデルを、同じ「物差し」で横並び比較した記録です。
モデルを差し替えるたびにこの手順 (`.claude/skills/model-eval`) を再実行し、本表を更新してください。

**測定データセットは v2(153 語 / 67 グループ / 11628 ペア)**。下の「ベンチマークの飽和対策」の通り、
最上位モデルが容易に満点へ達して分離できなくなった v1(74 語 / 2701 ペア)を難化したもの。
本ページの数値はすべて v2 での実測値です。

## 物差し

- **評価データ**: `tests/data/glossary.txt`(日本語の表記ゆれ + 日英中のクロス言語同義 + 難正例 + 難負例)
  - 同一グループ内ペア = 統合すべき(正例)、異グループ間ペア = 統合すべきでない(負例)
- **実行**: `cargo run --example benchmark -- <model> [threshold]`(指標ロジックは `src/eval.rs`)。
  ORT 非依存の Candle 勢は `cargo run --release --no-default-features --features candle --example candle_eval -- <model>`。
- **既定閾値**: `DEFAULT_THRESHOLD = 70.0`(CLI のデフォルト値)
- **モデル間比較は最適閾値で行う**: 固定の 70.0 での横並び比較は校正(`cos_baseline`)依存で各モデルの最適動作点を反映しないため意味が薄い。本記録では best-thr / clusterF1 真ピーク(=各モデルの最適閾値での値)で比較する
- 指標の意味:
  - **best-F1 / best-thr**: 分類 F1 を最大化する閾値とその値(各モデルを最適閾値で測った素ペア性能の上限)
  - **margin**: `正例min − 負例max`。0 に近いほど正例・負例の分離が良い(完全分離なら正)
  - **clusterF1(真ピーク)**: `normalize`(union-find の推移閉包込み)の実出力と正解グループのペア F1 = **実運用の挙動**。各モデルの最適閾値(1 刻みスイープのピーク)で比較する
  - **誤統合(false merges)**: `normalize` の推移閉包込みの実出力で、**異グループの語を同一クラスタにしてしまったペア数**(= データ破壊)。表記ゆれ統合では誤統合は取りこぼしより痛いため、clusterF1 が同程度なら誤統合が少ないモデルを選ぶ。**各モデルの真ピーク閾値で測る**
  - **最難負例 / 最難正例(閾値非依存・飽和対策の主軸)**: 誤統合しやすい順の負例ペア(= 適合率の境界。上位モデルほどスコアが**低い**)と、取りこぼしやすい順の正例ペア(= 再現率の境界。上位モデルほどスコアが**高い**)を上位 10 件表示する。**clusterF1 が上位で頭打ちになっても、ここのスコアは分離が続く**ため、最上位モデル同士の比較に使う
  - **per_text**: 1 用語あたり埋め込み時間(語あたりでデータ内容に依存しないため、単一セッションで測った値を流用)

## ベンチマークの飽和対策(データセット v2)

v1(74 語 / 26 グループ / 2701 ペア)は、最上位モデルが容易に上限へ達してモデル間の優劣を分離
できなくなった(= 飽和)。代表例として **Qwen3-Embedding-4B が v1 で clusterF1 0.964・P=1.000・誤統合 0 件**に
達しており、これ以上高性能なモデルが来ても差が出ない。そこで用語集を **v2: 153 語 / 67 グループ /
11628 ペア**へ拡張し、最上位モデルでも誤りうる「難所」を増やした。

### 何を難しくしたか

- **難正例(再現率の難所)**: 表層がほとんど重ならない多言語同義語を 10 グループ追加。略語・外来語・
  漢語・英語・中国語が入り混じり、文字の重なりに頼れない。例: `便所 / トイレ / お手洗い / restroom / 厕所`、
  `医者 / 医師 / ドクター / doctor / 医生`、`じゃがいも / ポテト / 馬鈴薯 / potato / 土豆`。
- **難負例(適合率の難所)**: 埋め込み空間で近接するが統合してはいけない語を単独グループで追加。3 種:
  - **共下位語**(同カテゴリの兄弟語): 季節 `春/夏/秋/冬`・色 `赤/青/黄色`・曜日・方角・兄弟・飲料
  - **紛らわしい異義語**(字面が近い別語): `科学/化学`・`保証/保障`・`関心/感心`・`公園/公演`
  - **対義語**(同じ話題で極性だけ反転し空間上で近い): `増加/減少`・`拡大/縮小`

### 飽和が解消された実証(v1→v2)

最上位 **Qwen3-4B は v1 で「完璧」(P=1.000・誤統合 0)だったが、v2 では実際に誤る**ようになった。
clusterF1 は依然トップだが頭打ちが解け、難負例 `保証⇔保障`(89.1)`厕所⇔台所`(80.9)を誤統合し、
難正例 `人参⇔carrot`(37.7)を取りこぼす。難所の連続スコア(最難負例/最難正例)で更に上のモデルとの
分離が測れる状態になった。他モデルも軒並み誤統合が増え、上位帯が広がった:

| モデル | v1 真ピーク | **v2 真ピーク** | v1 誤統合 | **v2 誤統合(真ピーク)** | v2 最難負例top |
|---|---:|---:|---:|---:|---|
| Qwen3-4B | 0.964 | **0.956** | **0** | **7** | 保証⇔保障 89.1 |
| Qwen3-0.6B | 0.748 | **0.764** | 1 | **10** | 減少⇔縮小 81.2 |
| bge-m3 | 0.725 | **0.699** | 1 | **7** | 月曜日⇔水曜日 89.0 |
| granite-278m | 0.705 | **0.682** | 24 | **28** | 兄⇔弟 91.5 |
| gte | 0.682 | **0.657** | 17 | **27** | 春⇔秋 83.1 |
| distiluse | 0.667 | **0.662** | 3 | **12** | 火曜日⇔水曜日 98.4 |

`兄⇔弟`・`月曜日⇔水曜日`・`保証⇔保障`・`減少⇔縮小` は多くのモデルが高スコアで誤統合する代表的な
難負例で、最上位モデル同士でもここのスコア差が優劣を分ける。

## 採用判断・所見

- **精度の絶対最高は Qwen3-Embedding-4B(Candle・精度上限枠)**。v2 でも真ピーク **0.956**(@70・P=0.963・
  R=0.948)で全モデルを大きく引き離す。v1 では P=1.000・誤統合 0 と「完璧」だったが、**v2 では誤統合 7 件**
  (`保証⇔保障` `厕所⇔台所` など)・難正例 `人参⇔carrot`(37.7)取りこぼしと、初めて明確な誤りが出た。
  弱点は速度/RAM(Candle CPU・f16・1 件ずつ前向き計算・≈3.9 秒/語・約 8GB RAM)。GPU・バッチ・オフライン
  等で速度を許容できる精度最優先用途向け(`--model qwen3-4b`、Apache 2.0)。
- **その中間が Qwen3-Embedding-0.6B(Candle・精度枠)**。真ピーク **0.764**(@71・P=0.926・誤統合 10)で、
  bge-m3(0.699)を精度で上回る。candle 0.9 の制約から 1 件ずつ前向き計算するため ≈200ms/語(bge-m3 の
  約 12 倍)。**ONNX 非依存環境(Candle 単独ビルドでは既定)の最有力**(`--model qwen3`)。
- **既定は bge-m3(精度と速度のバランス・ONNX で高速)**。真ピーク **0.699 は ONNX 勢で最高**、かつ
  **真ピーク時の誤統合 7 件は強モデル中で最小**(gte 27・granite-278m 28・distiluse 12・large 41)。
  ピーク時 P=0.939 と適合率も最高水準。「高 clusterF1 は誤統合増と引き換え」というトレードオフ
  (granite-278m は 0.682 だが誤統合 28)を破り、精度・安全性・サイズ(1.06GB)で gte(0.657 / 誤統合 27)を
  上回る。`Xenova/bge-m3` の fp16 単一 ONNX を読み込む(`--model bge-m3`、`cos_baseline=0.072`)。唯一の弱点は
  推論速度(gte の約 3 倍 ≈16ms/語)。
- **速度重視の代替は gte / 軽量代替は distiluse**。gte は真ピーク 0.657 で推論が bge-m3 の約 1/3(≈5ms/語)
  と軽快だが**誤統合 27 件**と多め。distiluse は真ピーク 0.662 を高適合率(P=0.895)・誤統合 12 件で達成し、
  **最小サイズ(0.54GB)・最速級(2.45ms)**。v2 では distiluse が gte を clusterF1・誤統合の双方で上回る。
  ただし distiluse は単漢字かな(`猫⇔ねこ`)の難正例を取りこぼす(最難正例 0.0)弱点がある。
- **clusterF1/再現率を最優先するなら granite-278m(選択肢)**。真ピーク 0.682(bge-m3 に次ぐ)で再現率
  R=0.593 が高いが、**誤統合 28 件**と「猫=犬」型のデータ破壊が多い。再現率最優先の用途に留める
  (`--model granite`)。
- **granite 系のその他 / e5 系 / paraphrase 系**は上位群に届かない。granite-311m-r2(0.652@87)・
  granite-107m(0.652@80)は高閾値ピークで誤統合も多い。e5-large(0.638)は E5 系で最高だが重い(13.6ms)。
  e5-small(0.561)は最軽量・最保守(ピーク時 P=0.853)で誤統合 14 件と少なく、保守性重視の選択肢。
  e5-base(0.514)は e5-small に劣後。paraphrase(0.591@89)・mpnet(0.567@86)は高閾値ピークかつ誤統合
  80 / 79 件と多い。
- **multilingual-e5-large-instruct は見送り(`--model e5-instruct`)**。真ピーク 0.645 で、**誤統合 75 件と
  全評価モデル中で最多**(指示プレフィックスでコサインが高帯域に寄り推移閉包が異義語まで巻き込む)。
  Candle で safetensors を直読みできるため ONNX 非依存環境の選択肢には残す。
- **snowflake-arctic-embed-l-v2.0 は見送り**。検索特化のため対称類似度で低調(真ピーク 0.495・@52)。
  素の bge-m3(0.699)が大差で上回り、検索特化の再学習が本タスクには逆効果と分かる
  (`--example benchmark -- arctic-l`)。
- **校正の含意**: 最適閾値が 70 から外れても、それは `cos_baseline` の校正で動かせるため優劣ではない。
  既定 70 へ寄せるための調整であって評価軸ではない。
